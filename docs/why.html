Why ProtoConsent

Consent banners don't work for users. We think purpose-based controls belong in the browser.

The problem with consent today

Every time you visit a website, you're asked to make privacy decisions through a consent banner. Accept all, reject all, or wade through dozens of vendor-specific toggles you didn't ask for. Most people click "accept" just to access the content. That's not informed consent — it's consent fatigue.

The result: regulators expect consent to be specific and freely given, but the mechanism is designed to exhaust users into compliance. Consent dialogs are fragmented per site, use inconsistent language, and are often built with dark patterns that nudge towards "accept all."

Meanwhile, the tools that exist sit at two unhelpful extremes. Content blockers work at the level of domains and filter lists — effective, but blunt. Consent management platforms (CMPs) work at the level of each individual website and its vendors — flexible for the site, but opaque to the user. There's nothing in between.

The missing layer

There is no simple, browser-level place where you can say: "on this site, I allow functional use and analytics, but never ads or advanced tracking" — and have that preference enforced consistently, without relying on each site to honour it.

ProtoConsent fills that gap. It's a browser extension that lets you define how different purposes of data use should be treated — per site and from a single interface. Instead of deciding vendor by vendor in each cookie banner, you set high-level profiles (Strict, Balanced, Permissive) and adjust individual purpose toggles (functional, analytics, ads, personalisation, third-party services, advanced tracking).

The key difference: ProtoConsent organises decisions around what your data is used for, not around which company is asking. This maps directly to how privacy regulation frames consent, and it's how most people actually think about their choices.

How it works, in practice

ProtoConsent is a Chromium extension (Firefox planned) that runs entirely on your device:

  • Per-site profiles — assign a general trust level (Strict, Balanced, Permissive) to any website, then refine specific purposes if you care about the details.
  • Network-level enforcement — the browser's own blocking APIs (declarativeNetRequest) prevent requests associated with disabled purposes from reaching remote servers. The popup shows per-purpose blocked counts and estimated performance savings.
  • Conditional GPC signal — sends a Sec-GPC header only when privacy-relevant purposes are denied for a site. Not a global toggle — a per-site, purpose-driven signal with legal weight under CCPA/CPRA.
  • Site declarations — websites can publish a .well-known/protoconsent.json file declaring their data practices. The popup displays this in a side panel with Consent Commons icons — no SDK required, just a static file.
  • Local-only storage — no backend, no account, no cloud sync. All your preferences stay in your browser.
  • Optional SDK — websites that choose to respect ProtoConsent can read a minimal, privacy-preserving signal from the browser ("is analytics allowed here?") without learning your identity or tracking you across sites.

When a site's embedded scripts try to contact analytics, ads, or tracking services, ProtoConsent checks whether that purpose is allowed for the current domain. If it's not, the request is blocked before it leaves your browser. If it is, it proceeds normally. No on-page scripts, no external servers, no new tracking mechanisms.

What ProtoConsent does not do

ProtoConsent is not a full ad blocker, not a VPN, and not a replacement for consent management platforms. It doesn't remove cookie banners from the page and doesn't claim to solve every privacy problem.

What it does is add a layer that didn't exist: a personal consent control panel in the browser, organised around purposes, that can work alongside the tools you already use. Think of it as complementary to your existing setup — content blockers, privacy-focused browsers, or CMP preferences — not a replacement for any of them.

Why now

Browser APIs have matured to the point where real enforcement is possible without injecting scripts into every page. Manifest V3 extensions, declarativeNetRequest, and the Storage Access API provide the technical foundation for purpose-based controls that are fast, safe, and sandboxed.

At the same time, regulators and standards bodies are actively discussing how browsers should mediate consent. ProtoConsent already sends a conditional Global Privacy Control (Sec-GPC) header — a signal with legal weight under California's CCPA/CPRA — only when privacy-relevant purposes are denied per site, not as a blunt global switch. The Permissions API, Storage Access API, and ongoing W3C privacy work all point in the same direction: the browser as a consent agent, not just a consent display.

ProtoConsent is a working implementation in that direction — open, inspectable, and aligned with where the web is heading.

Current status

The Chromium extension works and can be installed locally in developer mode. It provides purpose toggles for six categories, enforces choices via 40 000+ curated tracker domains and 1 200+ path-based rules organized in static rulesets by purpose, sends a conditional GPC signal, shows per-purpose blocked request counts with estimated performance savings, supports site declarations via .well-known/protoconsent.json with Consent Commons icons, and includes a JavaScript SDK (MIT) with TypeScript declarations for sites that want to read user preferences.

ProtoConsent is pending review in Chrome Web Store, Edge Add-ons, and Opera Addons. It is fully functional for local installation on any Chromium-based browser, with Firefox support planned.

Get involved

ProtoConsent is free and open source — GPL-3.0+ for the extension, MIT for the SDK, CC BY-SA 4.0 for documentation. Everything is developed in public on GitHub.

  • Publish a .well-known/protoconsent.json on your site to declare your data practices.
  • Integrate the SDK (MIT) to read user preferences from your pages.
  • Open an issue or pull request on GitHub with feedback, bug reports, or ideas.
  • Try the live SDK test on the project website or visit demo.protoconsent.org for a full-featured demo with site declaration, SDK and GPC signal check.
  • Reach out at contact@protoconsent.org for questions or collaboration.
